# test-authoring

## 目的
契約と回帰リスクに対して独自の検出価値を持つテストだけを作成、更新、整理する。
各テストが対象とする所有境界、検出する失敗、必要な入力と観測範囲を明確にし、最小のテスト集合で回帰を検出できる状態にする。
テストコードの設計と実装には $code-authoring-rules を併用する。

## フロー

### Phase 1: 契約と回帰リスクを固定する
依頼、合意済みの要求、変更前から有効な契約から、対象作業で守る結果と失敗条件を定める。
契約は、責務境界から観測できる結果、失敗条件、不変条件で表す。契約を決める根拠と、テストで検出する回帰を区別する。
利用例や受け入れ確認のシナリオは、その場面で保証する結果の根拠として扱う。明示されていない対象範囲、失敗条件、対応外条件までは導かない。
コード要素の存在、現在の実装手順、テストの書きやすさから契約を追加しない。
検証の準備、実行環境、観測手段に固有の条件を、対象契約へ持ち込まない。
実装やテストは、それ自身の期待結果を決める根拠にしない。同じ未確認の前提から作った実装、テスト、スキーマ、文書が一致していても、独立した契約根拠として数えない。

### Phase 2: 既存テストと所有境界を確認する
対象契約をすでに検証しているテストと、同じ失敗を検出するテストを探す。
テストは、契約の保証を所有する最も狭い責務境界を対象にする。
所有境界の正常完了時に保証が成立し、契約違反時にその境界で失敗することを観測する。
保証を所有しない境界では、先行する保証を再検証せず、その境界が新たに成立させる契約だけを検証する。
先行する保証が現在の境界で必要な対象、時点、成立範囲を覆わない場合は、その差分を契約として特定してから検証する。
既存テストで回帰を検出できる場合は、新しいテストを追加せず、必要なら既存テストを更新する。

### Phase 3: テストの形を決める

#### 入力と代表ケース
- 契約を分岐させる条件だけを入力に含め、それ以外は最小値または既定値にする。
- 同じ結果と失敗形態に属する入力は同値とみなし、一つの代表ケースで検証する。
- 境界値は、その境界で結果、失敗形態、または回帰リスクが変わる場合に選ぶ。
- 入力値やコード経路が異なるだけでは、別のテストを置かない。
- パラメータ化は、同じ契約と観測方法を保ったまま、診断可能性と読みやすさを維持できる場合に使う。

#### 観測範囲
- 出力、状態変化、失敗、外部への副作用のうち、対象契約を直接示すものだけを確認する。
- 内部の呼び出し回数、完全な呼び出し順序、中間状態は、それ自体が契約である場合だけ固定する。
- 禁止された副作用や境界到達は、その禁止を直接破ったときに失敗する観測で検証し、正常経路の内部手順全体を固定しない。
- 入力した値を同じ形で読み戻すだけの確認は、変換、正規化、検証、公開出力の契約を証明する場合に限る。
- 一つの整合した結果を示すために必要な値は同じテストで確認し、アサート数やコード要素数に合わせて分割しない。
- テスト名は、条件と保証する結果を表す。

#### 実行境界と信頼性
- 対象契約を検出できる最小の依存関係とセットアップで実行する。
- 実ファイル、外部プロセス、共有状態などは、その境界の振る舞いが検証対象に含まれる場合だけ使う。
- 一連の処理全体を通すテストは、全体を通すことでしか検出できない境界間の接続または変換の代表に限る。
- 共有セットアップは、各テストで異なる入力と対象契約を隠さない範囲で使う。
- 同じ契約条件では同じ判定結果を返すようにし、契約に含まれない時刻、乱数、実行順序、共有状態、実行環境を制御する。
- 非決定性そのものが契約に含まれる場合は、特定の実行結果ではなく、保証される範囲や不変条件を検証する。
- 出力全体を固定する比較は、出力全体が安定した契約であり、個別の観測へ分解すると重要な差異を失う場合に使う。
- 不安定なテストは、原因を解消するまで契約の証拠として数えない。隔離する場合は、既存の課題管理またはテスト注釈に、失われる証拠、暫定措置、必要な作業、復帰条件を残す。

### Phase 4: 必要性を判定する

#### 判定基準

| 判断 | 通過条件 |
| --- | --- |
| 契約 | テストが保証する結果と、その結果を決める独立した根拠を示せる。 |
| 検出価値 | そのテストを除いたときに検出できなくなる回帰を示せる。 |
| 既存テスト | 既存テストが同じ回帰を同等以上に検出しない、またはこのテストで置き換える理由がある。 |
| 所有境界 | 契約の保証を所有する最も狭い責務境界で、正常完了と契約違反を直接検証できる。 |
| 観測範囲 | 対象契約を直接示し、無関係な状態を固定しない。 |
| 実行境界 | 回帰を検出できる最小の依存関係とセットアップで実行できる。 |

すべてを満たす候補だけを選ぶ。
候補が複数ある場合は、契約、失敗形態、既存テストとの差、所有境界、観測範囲、実行境界を比較し、独自の検出価値を持つ候補だけを残す。
対象範囲を狭める条件または新しい失敗条件について、独立した根拠を示せない候補は選ばない。
基準を満たさない既存テストは、Phase 6 の整理対象にする。

#### 次の進行
作成または更新する候補がある場合は Phase 5 へ進む。
既存テストの整理だけが必要な場合は Phase 6 へ進む。
どちらも必要なく完了条件を満たす場合は Phase 7 へ進む。

### Phase 5: テストを作成または更新する

#### 新規テスト
選んだテストを 1 件ずつ扱う。
新しい振る舞いまたは不具合修正では、対象テストを実行し、意図した理由で失敗することを確認してから実装を修正する。
変更前から有効な契約へ検証を追加する場合は、失敗を作るために実装を変えず、契約違反と正常な結果を区別できる観測を置く。

#### 既存テストの更新
既存テストを更新する場合は、期待結果の根拠と現在の契約を確認する。

- 契約が変わっていないテストが失敗した場合は、テストを現在の実装へ合わせず、契約を破った実装を修正する。
- 合意済みの契約が変わった場合は、新しい契約と回帰リスクに合わせてテストを更新する。
- 契約を確定できない場合は、アサートを弱めず、今回の扱いと確認が必要なことを残す。

#### 局所検証
実装またはテストの変更後は対象テストを実行し、契約を満たして成功することを確認する。
この局所実行はテスト作成に含め、変更全体の受け入れ検証とは分ける。
局所検証後は Phase 6 へ進む。

### Phase 6: テスト集合を整理する

#### 整理基準
対象契約を検証するテスト群を読み直す。
次のいずれかに該当するテストは、独自の検出価値を確認し、統合または削除する。

- 保証する契約または検出する回帰を説明できない。
- 他のテストが同じ失敗を同等以上に検出する。
- 現在の実装手順、内部状態、値の保持だけを固定している。
- 検証の準備、実行環境、観測手段に固有の条件を対象契約として固定している。
- 先行する保証が現在の境界まで有効なのに、同じ条件を再検証している。
- 保証の所有境界が移り、旧境界の検証が残っている。
- 有効でなくなった契約を検証している。

削除前に、そのテストを除くことで検出できなくなる契約違反がないことを確認する。

#### 再判定
整理後に影響範囲のテストを実行し、残したテストが対象契約を検証できることを確認する。
未検証の契約または回帰リスクが残る場合は Phase 2 へ戻る。
契約の根拠または所有境界を見直す必要がある場合は Phase 1 へ戻る。
完了条件を満たす場合は Phase 7 へ進む。

### Phase 7: 作業状態を残す
作成、更新、削除したテスト、それぞれが検出する回帰、実行結果、未検証の契約を残す。
契約を確定できない場合は、確認が必要なことと、今回どう扱ったかを分ける。

## 完了条件
- 残した各テストについて、契約、検出する回帰、保証の所有境界と成立時点を説明できる。
- 同じ保証を検証するテストが重複していない。同じ条件の検証を複数の境界に残す場合は、対象、時点、成立範囲の差から、それぞれが所有する契約を説明できる。
- テストの入力、観測範囲、実行境界が、対象契約に必要な範囲に限定されている。
- 対象テストと影響範囲のテストが成功し、未実行の範囲が明らかになっている。
