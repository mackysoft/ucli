# XMLドキュメント執筆規約

## 全体ルール
- 本文は **English** で記述する。
- 命名の言い換えは書かない。命名以上の意図や情報がない場合は省略する。
- 曖昧語を避け、条件と境界を具体化する。
- 契約が存在する場合、`<param>` / `<returns>` / `<exception>` は省略しない。
- タグ本文の前後にスペースを入れる。
  - 例: `<param name="x"> text </param>`
- 責務境界を維持し、対象の責務外情報を書かない。

## `<summary>`
- 「何をするか」を誤解なく記述する。
- 仕様上の重要点を含める。
- 本文が複数行になる場合は各行を `<para>` で分割する。
- 命名以上の意図が書けない場合は省略する。

## `<param>` / `<typeparam>`
- 入力契約を記述する。
- 契約の対象:
  - 許容範囲
  - `null` 可否
  - 空文字、空配列、空コレクションの扱い
  - 単位
  - 前提条件
- 例外として、`cancellationToken` がキャンセル用途のみで追加契約を持たない場合は、`<param>` を省略する。

## `<returns>`
- 戻り値契約を記述する。
- 契約の対象:
  - `null` を返す条件
  - 空コレクションの意味
  - 順序、重複、安定性
  - スナップショットかライブか
- 非同期メンバは必要に応じて完了条件を記述する。

## `<exception>`
- 契約として送出し得る例外のみ列挙する。
- 各例外の発生条件を具体的に記述する。
- `CancellationToken` を受け取るメンバでは、キャンセル時の `OperationCanceledException` を記載しない。

## 補助タグ
- `<remarks>`: 補足が命名以上の意図で書ける場合のみ使う。
- `<example>`: 誤用が起きやすいAPIのみ最小例を記述する。
- `<para>`: 複数行の説明を分割するときに使う。
