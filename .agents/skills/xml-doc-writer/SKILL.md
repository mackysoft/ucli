# xml-doc-writer

## 目的
XMLドキュメントコメントで、対象APIの呼び出し契約を明文化する。

書く内容は、実装から確認できる入力制約、戻り値保証、例外条件、非同期完了条件に限る。
命名の言い換えだけのコメントは追加しない。

## フロー
### Phase 1: 対象を決める
公開面またはモジュール境界になるAPI（`public` / `protected` / `internal`）を優先する。
非公開メンバは、境界処理、変換、永続化、非同期、例外契約など、誤用リスクが高い場合だけ対象にする。

生成コード、実装詳細だけの局所処理、命名以上の契約を提供できない単純アクセサは対象にしない。

### Phase 2: 契約を読む
署名と実装から、入力制約、戻り値保証、副作用、例外条件、非同期完了条件を整理する。
曖昧な点は推測で埋めず、未確定として扱う。

例外は、契約として送出し得るものだけを対象にする。
`CancellationToken` を受け取るメンバでは、キャンセル時の `OperationCanceledException` を通常の例外契約として扱わない。

### Phase 3: タグを選ぶ
契約情報に対応するタグだけを選ぶ。

| タグ | 書く条件 | 本文に入れる情報 |
| --- | --- | --- |
| `<summary>` | 対象メンバに、命名だけでは伝わらない契約や責務がある。 | 呼び出し側が依存する役割、成立条件、重要な振る舞い。 |
| `<typeparam>` | 型パラメータに制約、用途、呼び出し側の前提がある。 | 型に求める性質、許容範囲、利用側が満たす前提。 |
| `<param>` | 引数に呼び出し側が守る入力契約がある。 | `null`、空、範囲、単位、所有権、寿命、順序などの条件。 |
| `<returns>` | 戻り値に呼び出し側が依存する保証がある。 | `null` 条件、空の意味、順序、重複、安定性、非同期完了条件。 |
| `<exception>` | 契約として送出し得る例外がある。 | 例外が発生する入力条件、状態条件、外部失敗条件。 |
| `<remarks>` | `<summary>` だけでは誤用を防げない補足がある。 | 副作用、永続化、キャッシュ、スレッド、性能、利用上の制約。 |
| `<example>` | 誤用が起きやすく、最小例が契約理解に効く。 | 契約を満たす最小の使い方。 |
| `<para>` | 一つのタグ本文に複数の説明単位がある。 | 一段落につき一つの契約または補足。 |

`cancellationToken` がキャンセル用途だけで追加契約を持たない場合、`<param>` は省略してよい。

### Phase 4: コメントを書く
タグ本文は `$writing` で文章前提を固定し、媒体はXMLドキュメントコメントとして扱う。
本文の言語は、ユーザー指定または既存のXMLドキュメントに合わせる。

命名の言い換えは書かない。
曖昧語を避け、条件と境界を具体化する。
タグ本文の前後にスペースを入れる。
対象の責務外情報を書かない。

### Phase 5: 点検し状態を残す
ドキュメントが実装と矛盾していないか確認する。
`<summary>` が動作を誤解なく示しているか確認する。
契約がある `<param>` / `<returns>` / `<exception>` が記載されているか確認する。
命名の言い換えだけの文を追加していないか確認する。

最後に、追加または更新したメンバ、対象にしなかったメンバと理由、未確定の契約情報を残す。
