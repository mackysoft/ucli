# code-authoring-rules

## 目的
ソースコードの実装、変更、レビューに共通する判断規則を定める。
コードを実行可能な振る舞いだけでなく、責務、契約、依存関係、設計意図を追える状態に保つ。
言語、フレームワーク、プロジェクト固有の規則は、この規則に重ねて適用する。

## 優先順位
1. プロジェクト固有の明示的な契約と規則を守る。
2. 振る舞い、不変条件、失敗条件を正しく保つ。
3. 責務、状態、依存関係、副作用の境界を明確にする。
4. 同じ責務と契約を持つ実装を一つに収束させる。
5. 将来の変更判断に必要な設計意図を、コードの近くに残す。

## 設計をコードへ写す

### 責務の境界
実装前に、変更が扱う責務と、その責務を所有する型、関数、モジュールを定める。
責務は、入力、出力、保有する状態、不変条件、副作用、依存関係、失敗条件、寿命から判断する。
所有者を定められない場合は、呼び出し元、データの流れ、依存方向、寿命を確認し、境界を定めてから実装する。
同じ理由で変更される振る舞いを同じ責務へまとめ、異なる変更理由を持つ振る舞いを分ける。
上位方針と下位の実装詳細を同じ責務へ押し込まない。
名前と配置は責務と契約を表し、契約、その実装、検証の対応関係を追える状態にする。
異なるレイヤーや関係の薄い責務を同じ階層へ混在させない。
既存コードは契約と慣習を知る根拠として使う。
責務境界と契約が成立している既存設計には整合させ、命名と配置は周辺コードで確立している慣習に合わせる。
弱い責務境界を正しい設計として追認しない。

### 契約の所有
入力、出力、不変条件、失敗条件には、それを保証する境界を一つ定める。
呼び出し側は事前条件を満たし、呼び出し先は自身が所有する検証と事後条件を保証する。
呼び出し側で検証するのは、事前条件を満たすために入力を変換する場合、呼び出し側固有の不変条件を守る場合、検証結果に応じて回復またはエラー分類を行う場合に限る。
契約違反は、その条件を所有する境界で失敗させる。
成立済みの契約を下流で推測し直したり、同じ条件を同じ目的で複数の境界に実装したりしない。
利用側が必要としない自由度を公開契約へ持ち込まない。

### 依存関係と副作用
上位方針を下位の実装詳細へ直接依存させない。
利用側に不要なメンバ、状態、依存を公開しない。
計算、判定、変換と副作用を分け、副作用の対象、実行時期、成功時と失敗時に残る状態を追える境界へ閉じる。
副作用を持つ処理を、純粋な処理に見える名前や契約で公開しない。
テスト、ツール、移行作業の都合を、本体コードの依存方向や公開契約へ逆流させない。
中核となる判断は外部状態から分離し、契約、不変条件、境界条件、失敗条件を単体テストで検証できる形にする。

## 実装を収束させる

### 既存実装の利用
新しい処理を追加する前に、名前だけでなく、責務、入力、出力、失敗条件から既存の型、関数、ヘルパを探す。
同じ責務と契約を持つ実装がある場合は、その実装を使用する。
変更が必要な場合は、既存実装と利用側を更新し、同じ処理を別の場所へ追加しない。
既存実装と分けるのは、契約または所有責務が異なり、独立した変更単位として維持する必要がある場合に限る。
偶然形が似ているだけで意味が異なる処理は、無理に共通化しない。

### 意味を持つ構造
型、関数、変数、分岐、例外、抽象化には、振る舞い、契約、責務、状態、判断のいずれかを表現または分離する役割を持たせる。
呼び出しを委譲するだけの層は、独自の契約または責務境界を持つ場合に限る。
到達しない分岐、常に同じ結果になる条件、使われない例外処理、役割を失った入口を残さない。

### 変更範囲の整合
新しい契約を導入するときは、既存契約の置換、既存契約から独立した契約の追加、明示された互換期間の併存のどれに当たるかを定める。
既存契約を置き換える場合は、利用側と検証を新しい契約へ合わせ、旧契約だけを支える入口を削除する。
独立した契約として追加する場合は、既存契約と異なる利用要件と責務を説明できる状態にする。
互換性が明示的な要件でない限り、旧契約を残すための別名、オーバーロード、省略可能な入力、既定値、フォールバックを追加しない。
互換性を維持する場合は、維持する契約と移行条件を明確にし、互換処理を一つの境界へ局所化する。
変更箇所だけでなく、同じ責務を持つ周辺の型、関数、利用側を確認し、一つの設計へ揃える。

## 設計意図を残す

### 処理群の責務
長い処理や複数段階の処理では、各処理群が上位処理の中で担う責務を、その直前に短く記す。
コメントは行ごとの処理を言い換えるのではなく、そのまとまりが何を成立させるかを示す。
処理群が独立した変更理由や契約を持つ場合は、コメントで束ねず、別の責務へ分ける。

### 理由と制約
コードの構造だけでは復元できず、将来の変更判断に影響する理由、制約、不変条件、成立条件を、対象コードの近くに記す。
自然に見える変更や別の実装が成立しない場合は、短い `NOTE` で現在有効な理由、守る条件、変更した場合の影響を残す。
選ばなかった方法を列挙するのではなく、その方法が成立しない現在の条件を書く。

### コメントとドキュメント
公開契約の目的、利用方法、入力、出力、失敗条件は、言語が提供するドキュメント機構で表す。
実装コメントには、契約を実現するための判断理由と、コードだけでは表せない制約を書く。
コメントとドキュメントの文面には $writing を適用する。
コメントは対象となるコードの近くに置き、コード変更によって役割を失った記述は、同じ変更で更新または削除する。

## 整合の確認
対象箇所だけでなく、その責務を含む型、関数、モジュールを通して読む。
各責務の所有者を説明でき、契約の保証箇所が重複せず、同じ意味の実装が一つに収束していることを確認する。
処理群の責務と非自明な設計理由がソースコードから追え、コメントとドキュメントが現在の実装に一致している状態にする。
