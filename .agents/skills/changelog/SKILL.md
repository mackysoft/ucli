# 目的
- 変更素材を、読者向けの `CHANGELOG.md`、リリースノート、PR本文、コメント用差分ログへ変換する。
- コミットログ、開発日誌、PR一覧、実装メモではなく、読者が利用、移行、運用、調査の判断に使える変更説明を書く。

# 使うタイミング
- `CHANGELOG.md` にエントリを追加または編集する。
- リリースノートを書く。
- PR本文やレビューコメントに、その差分の Changelog セクションを書く。
- コミット、Issue、PR、diff、既存履歴から人間向けの変更要約を作る。

# 使わないタイミング
- バージョン番号、リリース可否、公開手順、承認フローだけを決める。
- Conventional Commits、Semantic Versioning、パッケージ公開手順そのものを整理する。

# 契約
- ユーザーの明示指示を最優先する。
- 既存の changelog やリリースノートがある場合は、見出し階層、日付形式、カテゴリ順、箇条書き、リンク形式、文体を維持する。
- 変更内容、バージョン番号、リリース日、Issue/PR番号、移行手順、互換性影響、セキュリティ影響を推測で作らない。
- バージョンまたは日付が不明な場合は、既存形式に合わせて `Unreleased` に入れるか、不明な値を省略する。
- 対象読者を推定できず、含める情報の粒度が大きく変わる場合は、書く前に対象読者を確認する。
- 確認できない場合は、既存文書の粒度とプロジェクト種別から最も保守的な読者を仮定し、推測した前提を出力内で短く明示する。
- 対象文書の言語に合わせる。既存文書が英語なら英語、日本語なら日本語で書く。

# 手順
## 1. 素材と既存形式を確認する
次の順に確認する。

1. ユーザー指示。
2. 既存の `CHANGELOG.md`、リリースノート、PR本文、コメントの形式。
3. 対象差分、コミット、Issue、PR、マイルストーン、関連議論。
4. プロジェクト種別と公開面。

既存形式から、見出し階層、カテゴリ名、バージョン表記、日付表記、リンク表記、文の長さ、情報粒度を読み取る。

## 2. 素材の信頼度を扱う
PRタイトル、Issue本文、commit message、レビューコメントは変更意図の手がかりとして扱う。
最終的な変更事実は、可能な限り diff、merged content、既存仕様、実際の公開面から確認する。

PRやcommitにある「改善」「高速化」「安全化」「完全対応」のような効果表現は、根拠が確認できる場合だけ changelog に反映する。
根拠が弱い場合は、観測できる変更事実に書き換える。

## 3. 読者を決める
読者は、ユーザー指示、既存履歴の粒度、プロジェクト種別から推定する。

- ライブラリ / SDK: API利用者、組み込み担当者、メンテナー。
- CLI / 開発ツール: コマンド利用者、自動化スクリプトの作成者、メンテナー。
- アプリケーション: エンドユーザー、運用担当者、サポート担当者。
- 内部基盤 / インフラ: 開発者、運用担当者、リリース担当者。
- 低レイヤーのフレームワーク / プロトコル: 挙動差分や互換性を読む技術者。

読者向け changelog では、観測可能な挙動、ワークフロー、API、コマンド、設定、互換性、移行、セキュリティ、運用影響を優先する。
メンテナー向けでも、内部変更はデバッグ性、拡張性、性能、信頼性、契約、運用リスクが変わる場合だけ含める。

## 4. 変更を選別する
コミットやPRを1対1で写さない。

- 1つの読者可視な変更を複数コミットが実現している場合は、1エントリへ統合する。
- 1つのPRに複数の読者可視な変更がある場合は、複数エントリへ分割する。
- 対象読者に意味がない機械的リファクタリング、テストのみの変更、整形、内部移動、足場作成、依存更新は、挙動、互換性、性能、セキュリティ、信頼性、運用性、公開契約が変わらない限り除外する。

素材に次が含まれる場合は、エントリまたは補足で明示する。

- 破壊的変更。
- 非推奨化。
- 削除。
- 必要な移行手順。
- 挙動または既定値の変更。
- API、コマンド、オプション、スキーマ、プロトコル、ファイル形式、出力形式の変更。
- セキュリティ影響。
- データ損失、互換性、運用上のリスク。
- 既知の制約や未対応範囲。
- 利用者に見える修正。
- プラットフォーム、ランタイム、形式、連携先の追加または削除。

破壊的変更、削除、移行手順、セキュリティ影響を曖昧な表現で隠さない。

## 5. 出力先に合わせて書く
### CHANGELOG.md
既存形式が無い場合は、Keep a Changelog を基準にする。

```md
## [Unreleased]

### Added

- Added ...

### Changed

- Changed ...

### Deprecated

- Deprecated ...

### Removed

- Removed ...

### Fixed

- Fixed ...

### Security

- Fixed ...
```

リリース済みバージョンを書く場合は次を標準形にする。

```md
## [x.y.z] - YYYY-MM-DD
```

最新バージョンまたは `Unreleased` を上に置く。
既存 changelog が空カテゴリを残していない限り、空カテゴリは書かない。

### PR本文またはコメント用エントリ
短いセクションにまとめる。

```md
### Changelog

- Added ...
- Fixed ...
```

項目数が少ない場合はカテゴリ見出しを省略し、十分な件数がある場合だけカテゴリ分けする。

### リリースノート
リリース全体の目玉がある場合だけハイライトを置く。
詳細は変更種別またはユーザー影響ごとにまとめる。
説明的な文章にしてよいが、開発文脈ではなく読者が理解できる成果と影響を書く。

# 変更カテゴリ
既存文書に分類がある場合はそれに従う。
無い場合は Keep a Changelog の分類を使う。

- `Added`: 新機能、コマンド、API、オプション、対応対象、利用可能になった能力。
- `Changed`: 挙動、意味論、既定値、出力、検証、ワークフロー、性能、互換性の変更。
- `Deprecated`: まだ使えるが、今後の削除や非推奨が決まったもの。
- `Removed`: 削除された機能、API、コマンド、オプション、挙動、対応対象。
- `Fixed`: 不具合修正、正確性修正、信頼性修正、回帰修正。
- `Security`: 脆弱性修正、強化、資格情報の扱い、露出低減。

既存文書が `Performance`、`Documentation`、`Migration`、`Breaking`、`Renamed` などを使う場合は、その形式に合わせる。
既存形式が無い場合、rename は `Changed`、性能改善は不具合修正でない限り `Changed`、ドキュメントのみの変更はドキュメントが利用者に見える成果物である場合だけ含める。

破壊的変更は該当カテゴリ内で明示する。

```md
### Changed

- **Breaking:** Changed the configuration file format to reject unknown fields.
```

# Security エントリ
Security impact がある場合は、ユーザーまたは既存リリース方針が示す開示粒度に従う。
未公開の脆弱性詳細、悪用手順、影響範囲を推測で書かない。
依存更新は、既知の脆弱性対応、資格情報保護、露出低減などの読者影響が確認できる場合だけ `Security` に入れる。

# エントリ文体
- 1エントリには、1つの読者向け判断単位だけを書く。
- 同じ機能、移行、互換性、運用判断に属する詳細は、1エントリ内にまとめてよい。
- 無関係な変更、別カテゴリの変更、別の移行判断が必要な変更は分ける。
- 変更対象、変更内容、読者への影響を一読で分かる順序にする。
- 英語では `Added`、`Changed`、`Deprecated`、`Removed`、`Fixed`、`Renamed`、`Improved`、`Documented` などの変更動詞から始める。
- 日本語では既存文体を優先し、対象と変更内容が冒頭で分かる表現にする。
- 「改善」「対応」「調整」のような広い語だけで終えない。何がどう変わったかを書く。
- 確認していない効果を断定しない。性能、安定性、セキュリティ、互換性の効果は根拠がある場合だけ書く。
- 後で参照する必要のない内部名、ブランチ名、コミットハッシュ、担当者名を出さない。
- 読者が理解するために必要な固有名、API名、コマンド名、設定名は省かない。

避ける例:

```md
- Refactor parser internals.
- Build profile parser cleanup.
- PR #123.
- Implement FooService.
- Update dependencies.
- Improved various behaviors.
```

読者向けに書き換える例:

```md
- Changed build profile validation to reject unknown nested fields.
- Fixed build profile loading when the profile path contains spaces.
- Added support for resolving Unity Build Profile assets during build input selection.
```

# 文章品質
- 一つのエントリで複数の話題を混ぜない。
- 同じ変更を言い換えて繰り返さない。
- 読者が判断に使わない作業手順や実装都合を書かない。
- 情報を増やさない前置きや空句を使わない。
- PR、実装作業、担当者視点を主語にした説明で終えない。
- 誤読されやすい変更は、何が変わらないかも短く書く。
- 制約、未対応範囲、移行条件は、確認できている範囲だけ具体的に書く。

# Issue / PR参照
既知の Issue または PR に基づく場合は、エントリ末尾に参照を置く。

同一 GitHub リポジトリ内の文書では短い参照を使う。

```md
- Fixed daemon startup diagnosis when Unity exits before endpoint registration. (#123)
```

外部向けリリースノートやリポジトリ外文書では、対象文書の形式に合わせて Markdown リンクまたはURLを使う。

```md
- Fixed daemon startup diagnosis when Unity exits before endpoint registration. ([#123](https://github.com/org/repo/pull/123))
```

複数参照は、そのエントリの根拠として有益な場合だけ付ける。
番号が不明な場合は参照を作らない。

# 既存CHANGELOGの編集
1. 既存の見出し階層、リンク形式、バージョン形式、日付形式、カテゴリ順、箇条書き、参照形式を維持する。
2. ユーザーが対象バージョンを指定しない場合は `Unreleased` に追加する。
3. 最新のリリースまたは `Unreleased` を上に置く。
4. 依頼が無い限り、過去リリースを並べ替えない。
5. 依頼が無い限り、過去エントリを書き換えない。
6. 既存の changelog 規約を黙って変更しない。

# Definition of Done
- エントリが読者向けであり、コミットメッセージの言い換えに留まっていない。
- 対象文書の形式、言語、文体に合っている。
- 変更カテゴリが適切である。
- 1エントリが1つの読者向け判断単位に対応している。
- 破壊的変更、非推奨化、削除、移行、セキュリティ影響が必要に応じて明示されている。
- バージョン、日付、Issue/PR参照を推測で作っていない。
- 素材に含まれる効果表現を過大表現にせず、確認できる変更事実に基づいている。
