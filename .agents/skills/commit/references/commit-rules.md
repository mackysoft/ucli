# Commit Rules

## 目的
- 1コミットに複数作業単位を混在させない。

## type の基本対応
- 新機能: `feat`
- バグ修正: `fix`
- 振る舞いを変えない再構成: `refactor`
- テスト変更: `test`
- ドキュメント変更: `docs`
- CI設定・ワークフロー変更: `ci`
- 上記に当てはまらない変更: `chore`

## scope 推定ルール
1. 単一モジュールの変更なら、そのモジュール名を使う。
2. 複数モジュールにまたがるが同一ドメインの変更なら、ドメイン名を使う。
3. 複数ドメイン横断で共通基盤の変更なら、`core` を使う。

## 件名ルール
1. 件名は `type(scope): summary` 形式にする。
2. `summary` は変更結果を具体的に表す1文にする。

## 分割ルール

**原則：1コミット = 1意図**

1. 異なる `type` は必ず別コミットにする。
2. 同じ `type` でも `scope` が別なら分割を優先する。
3. 同じ `type` かつ同じ `scope` でも、意図（目的）が複数あるなら分割を優先する
4. summaryに`and`などの接続が入る場合は分割を優先する。
5. 生成物のみの更新は独立コミットにする。
6. `docs` は機能変更と分離する。
7. 分割不能なら停止し、方針確認を行う。
