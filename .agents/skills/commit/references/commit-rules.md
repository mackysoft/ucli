# Commit Rules

## 目的
- 1コミットに複数作業単位を混在させない。
- コミット本文を、将来の保守者、レビュー担当者、障害調査者が変更意図を追跡できる説明にする。

## type の基本対応
- 新機能: `feat`
- バグ修正: `fix`
- 振る舞いを変えない再構成: `refactor`
- テスト変更: `test`
- ドキュメント変更: `docs`
- CI設定、ワークフロー変更: `ci`
- 上記に当てはまらない変更: `chore`

## scope 推定ルール
1. 単一モジュールの変更なら、そのモジュール名を使う。
2. 複数モジュールにまたがるが同一ドメインの変更なら、ドメイン名を使う。
3. 複数ドメイン横断で共通基盤の変更なら、`core` を使う。

## 件名ルール
1. 件名は `type(scope): summary` 形式にする。
2. `summary` は変更結果を具体的に表す1文にする。
3. `summary` は「対応」「改善」「調整」だけで終えず、何がどう変わったかを示す。

## 分割ルール

**原則：1コミット = 1意図**

1. 異なる `type` は必ず別コミットにする。
2. 同じ `type` でも `scope` が別なら分割を優先する。
3. 同じ `type` かつ同じ `scope` でも、意図（目的）が複数あるなら分割を優先する
4. summaryに`and`などの接続が入る場合は分割を優先する。
5. 生成物のみの更新は独立コミットにする。
6. `docs` は機能変更と分離する。
7. 分割不能なら停止し、方針確認を行う。

## 本文の読者
- コミット本文の読者は、将来の保守者、レビュー担当者、障害調査者とする。
- 内部実装の説明は、変更理由、設計判断、調査に必要な範囲で書いてよい。
- 本文は変更理由、変更事実、影響だけに絞る。担当者やブランチ名など、変更結果の理解に不要な情報は書かない。

## 本文ルール
1. 本文は `Why`、`What Changed`、`Impact` の3区分で書く。
2. `Why` には、変更が必要になった背景、解決したい問題、守りたい契約を書く。
3. `What Changed` には、diff から確認できる変更事実を書く。
4. `Impact` には、利用者、機能、互換性、運用、テスト、保守性への影響を書く。
5. 1つの箇条書きには、1つの保守判断単位だけを書く。
6. 同じ設計判断、契約、移行、運用判断に属する詳細は、1つの箇条書きにまとめてよい。
7. 無関係な変更、別の判断が必要な変更、別の `type` に属する変更は分ける。
8. 性能、安定性、セキュリティ、互換性の効果は、根拠が確認できる場合だけ書く。
9. 根拠が弱い効果表現は、観測できる変更事実に書き換える。
10. 誤読されやすい変更は、何が変わらないかも短く書く。

## 素材の扱い
- Issue本文、PR本文、commit message、レビューコメントは変更意図の手がかりとして扱う。
- 変更事実は、可能な限り diff、既存仕様、テスト、公開API、実装から確認する。
- 「改善」「高速化」「安全化」「完全対応」のような効果表現は、そのまま本文に写さない。
- 「リファクタリングした」「実装した」「対応した」だけで終えず、どの構造、契約、挙動、保守判断が変わったかを書く。
