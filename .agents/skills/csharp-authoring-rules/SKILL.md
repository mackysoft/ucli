# csharp-authoring-rules

## 目的
C# の実装、修正、レビューで適用する設計と実装のルールを定める。
変更前後に、責務、契約、依存、例外、非同期、寿命管理、現在の C# の書き方、テスト容易性を確認する。

## 優先順位
1. リポジトリ固有の明示規約を優先する。
2. 責務境界と契約を保っている既存設計がある場合は、その設計と矛盾しない形で変更する。
3. 既存コードが責務境界や契約を崩している場合は、現状を正当化せず、変更範囲で改善できる境界を選ぶ。

## ルール

### 責務境界
一つの型、関数、モジュールに複数の変更理由を混在させない。
上位方針と下位実装の詳細を同じ型へ押し込まない。
新しい処理を追加するときは、既存責務へ入る変更か、新しい責務として分ける変更かを判断する。
関係の薄い型を同じ階層へ置かない。

### 依存関係
上位方針を下位詳細へ直接依存させない。
利用側に不要なメンバや依存を公開しない。
既存の安定した契約を壊さず、拡張点で振る舞いを追加できる形を優先する。
テストやツールの都合を本体コードの依存方向へ逆流させない。

### 公開契約
公開 API は、呼び出し側へ約束する入力、戻り値、例外条件、不変条件を持つ。
呼び出し側が使わない自由度を公開 API に残さない。
契約を変える変更では、利用側の期待とテストを同じ変更範囲で確認する。
古い呼び出し方を残すためだけの互換コードを追加しない。

### 重複
同じ業務ルール、判定条件、変換規則、検証契約を複数箇所に重複実装しない。
既存の型、関数、ヘルパを確認してから新規実装する。
偶然形が似ているだけで意味が異なる処理は、無理に共通化しない。

### 意味を増やさないコード
意味を増やさない型、関数、分岐、変数、例外、ラッパーを残さない。
呼び出しを横流しするだけの層は、契約、責務境界、テスト境界のどれかを増やす場合だけ置く。
変更後は、追加したコードだけでなく、周辺の型やメンバがまだ意味を持つかを確認する。
到達しない分岐、常に同じ結果になる条件、使われない例外処理は削除する。

### 現在の C# の書き方
プロジェクトが対応する C# バージョン、ターゲットフレームワーク、実行環境を確認する。
対応範囲で使える現在の C# の書き方と標準ライブラリは、古い定型コードより優先して検討する。
書き方を選ぶ目的は、契約を明確にする、責務を小さくする、割り当てを減らす、例外条件を減らすことに置く。
慣れているだけの古い書き方を、対応範囲でより適切な書き方がある場合に選ばない。
新しい書き方を使うことで責務境界、可読性、対応する実行環境が崩れる場合は使わない。
処理ブロックでは波括弧を省略しない。

### 例外と入力検証
例外は契約として設計する。
入力検証は早期に行う。
例外を通常の制御フローに使わない。
例外を握りつぶさない。
呼び出し側が回復できる条件と、プログラムの契約違反を混同しない。
例外を投げる場合は、呼び出し側が原因を特定できる情報を残す。

### null と未設定状態
`null` を有効な状態として扱う場合は、その意味を契約として明確にする。
未設定、存在しない、解決できない、空である状態を同じ値へ押し込まない。
初期化の遅延で `null` を使う場合は、初期化責務と利用開始条件を分ける。
呼び出し側が必ず渡すべき依存を、省略可能な引数や暗黙の既定値にしない。

### 非同期 API と CancellationToken
`CancellationToken` は、キャンセル可能な操作の契約として扱う。
オブジェクトやサービス全体の状態として保持しない。
非同期 API は、実際にキャンセル可能な非同期処理を開始または委譲する場合に `CancellationToken` を受け取る。
下位 API がキャンセル可能であればトークンを渡す。
キャンセル不能な下位 API に合わせるためだけに `CancellationToken` を追加しない。
短い同期関数、単純な変換、即時に終わる検証、通常のプロパティ取得のために `CancellationToken` を追加しない。
長時間実行される同期処理、CPU 計算中心の処理、反復処理、探索、解析、バッチ変換で呼び出し側が中断を要求する必要がある場合は、同期関数でも `CancellationToken` を受け取ってよい。
その場合は、キャンセル確認を反復境界、処理単位の境界、外部リソース境界に置く。
同期処理を、実処理が同期のまま `Async` 化しない。
上位 API が非同期であることを理由に、下位の同期関数へ `CancellationToken` を機械的に伝播しない。
非同期関数では同期的な待機を行わない。
`ThrowIfCancellationRequested()` は、キャンセルを契約として観測する境界に置く。
短い同期関数の呼び出し直前、直後に機械的なキャンセル確認を置かない。

### 依存性注入
依存は原則としてコンストラクタで受け取る。
利用側に不要な依存を強制しない。
単純なラッパーでコードの複雑性を上げない。
生成、設定、実行の責務を一つの型へ集めない。
外から注入されるべき依存を、フォールバック用コンストラクタや既定引数で内部生成しない。
既定実装の生成は、依存関係を組み立てる起点または明確な生成責務を持つ型で扱う。

### 副作用の境界
副作用は、呼び出し元、対象、タイミングが追える場所へ閉じる。
計算、判定、変換の中に I/O、時刻、乱数、グローバル状態の変更を混在させない。
副作用を伴う処理は、成功時と失敗時に残る状態を契約として扱う。
副作用の隠れたヘルパを、純粋な処理に見える名前で公開しない。

### 寿命管理
`IDisposable`、`IAsyncDisposable`、ストリーム、ハンドル、タイマー、購読、`CancellationTokenSource` などを所有する型では、所有者と解放責務を明確にする。
例外時の解放も契約として扱う。
外から借りたリソースを、自分が所有しているように破棄しない。
イベント、通知、購読、リアクティブな通知は、生成元、購読元、解除元が対応する形にする。
購読解除が必要な場合は、解除責務を呼び出し側へ暗黙に押し付けない。
イベントハンドラに、通知の橋渡しを超えるアプリケーション方針を混在させない。
通知順序、重複通知、再入、例外伝播が契約に影響する場合は明示する。

### 不変条件
型が成立させる条件は、コンストラクタ、Factory、更新メソッドの境界で守る。
不正な中間状態を公開 API から観測できる形にしない。
複数メンバの組み合わせで成立する条件は、一つの責務として扱う。
検証と更新を分ける場合は、検証後に条件が変わらない範囲を明確にする。

### 命名と配置
命名は既存コードに合わせる。
名前は役割、契約、責務を表すものにする。
一つのファイルに複数のトップレベル型を置かない。
機能実装は、契約、実装、検証の対応関係が追える配置にする。
レイヤー境界を跨ぐ責務を同一ディレクトリへ混在させない。

### メンバ配置
フィールドは責務と性質ごとにまとめる。
ドキュメントコメント、属性、責務の切れ目があるメンバの前には空行を置く。

### コメントと XML ドキュメント
コメントは、コードから読めない背景、契約、制約を書く。
処理内容をなぞるだけのコメントは書かない。
公開 API または重要な契約には XML ドキュメントで入力、戻り値、例外条件を明示する。
背景が重要な処理には、短い `NOTE` コメントで意図を残す。

### テスト容易性
副作用は境界へ寄せる。
ロジックはユニットテスト可能な単位へ分ける。
テストのためだけに公開 API や本体設計を歪めない。
契約、境界条件、失敗条件をテストで確認できる形にする。
意味を増やさないラッパーを、テストのための差し替え口として追加しない。

### 割り当てとデータ構造
データ構造は、操作の計算量、割り当て回数、局所性を考慮して選ぶ。
頻繁に呼ばれる処理では、不要なコレクション生成、文字列生成、クロージャ生成を避ける。
配列、文字列、バッファの一部を扱うときは、不要なコピーを作らず、対応範囲で `Span<T>`、`ReadOnlySpan<T>`、`Memory<T>`、`ReadOnlyMemory<T>` を検討する。
`Span<T>` などの寿命、所有権、非同期境界に制約がある型は、その制約を満たせる範囲で使う。
割り当て削減のために責務境界や可読性を壊さない。

## 適用結果
変更後またはレビュー後に、適用した主要ルール、削除した意味を増やさないコード、未確認事項を残す。
