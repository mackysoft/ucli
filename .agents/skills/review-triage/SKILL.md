# review-triage

## 目的
レビュー所見を対象成果物、要求、契約、関連資料と照合し、妥当性と今回の対象との関係を分けて判断する。
重複した所見を統合し、各所見の根拠、対象との関係、判定、優先度、対応方針、未確定事項を整理する。

## 参照
- 出力テンプレート: [references/triage-report-template.md](references/triage-report-template.md)

## フロー
### Phase 1: 所見と対象を固定する
- 依頼と参照資料から、今回の目的、受け入れ条件、対象範囲を固定する。
- 所見の出所、対象成果物、変更内容、根拠資料を整理する。
- 所見ごとに一意な `ID` を付け、出所と対象成果物への反映状態を残す。
- 入力だけで対象を確定できない場合は、利用可能な作業領域と資料を探索して対象成果物と関連資料を特定する。

### Phase 2: 重複を統合して根拠を集める
- 同じ問題、契約違反、または判断を指す所見を 1 件へまとめ、最も具体的な記述を代表として残す。
- 根拠が不足する点は推測で補わず、確定に必要な情報を残す。

#### 根拠の役割

| 役割 | 扱い |
| --- | --- |
| 期待を決める根拠 | 依頼、合意済みの要求、受け入れ条件、明示された仕様から、期待する結果と対象範囲を決める。 |
| 変更前から有効な契約 | 期待を決める根拠と矛盾しない範囲で、維持すべき契約として扱う。 |
| 現状を示す情報 | 成果物の内容、構造、振る舞い、影響範囲の確認に使い、それだけで正しい期待とはみなさない。 |
| 今回の作業で生じた情報 | 提案内容と影響の確認に使い、その作業の必要性を単独では証明しない。 |
| 推論 | 仮説として扱い、期待を決める根拠、有効な契約、または明確な失敗形態による裏づけを求める。 |

情報は形式ではなく役割で扱う。期待する結果が衝突する場合は、期待を決める根拠、変更前から有効な契約の順に優先し、現状や今回の作業だけで期待を上書きしない。
同じ未確認の前提から派生した複数の成果物は、一つの情報源として扱う。
好みに基づく所見は、適用される規約、一貫して使われている規則、または明確な失敗形態で裏づけられる場合だけ採用候補にする。

### Phase 3: 妥当性と対象との関係を判定する

#### 対象との関係

| 関係 | 基準 |
| --- | --- |
| `直接` | 今回の変更が生じさせた問題、または今回の成果物が受け入れ条件や有効な契約を満たしていない問題である。 |
| `必須の前提` | 最小限の対応を行わなければ今回の目的を成立させられず、その依存関係と必要範囲を説明できる。 |
| `隣接` | 今回の対象と関連するが、対応しなくても今回の目的と受け入れ条件は成立する。 |
| `対象外` | 今回の目的、変更、受け入れ条件との因果関係または契約上の関係がない。 |
| `未確定` | 判断に必要な要求、契約、因果関係が不足し、対象との関係を確定できない。 |

対象との関係は、今回の目的、受け入れ条件、有効な契約、変更による因果関係から判断する。

- 同じ成果物、場所、主題、領域に属することだけでは、`直接` または `必須の前提` と判断しない。
- `必須の前提` は、対応しなければ今回の目的を成立させられず、必要な対応を最小範囲に限定できる場合に使う。
- 問題の重大性、修正の容易さ、改善効果は優先度の根拠にはなるが、対象との関係を変えない。
- レビューや検証で問題を発見した事実だけでは、その問題を今回の要求へ加えない。
- 検証失敗は、現在の変更、変更前から存在する問題、検証手段、実行環境を切り分けてから対象との関係を判断する。

#### 判定

| 判定 | 基準 |
| --- | --- |
| `採用` | 所見が妥当で、対象との関係が `直接` または `必須の前提` である。 |
| `却下` | 根拠と矛盾する、誤読に基づく、または対応を必要とする根拠がない。 |
| `保留` | 所見は妥当だが、対象との関係が `隣接` または `対象外` である。 |
| `要確認` | 要求、契約、意図、前提が不足し、妥当性を確定できない、または対象との関係が `未確定` である。 |

#### 優先度

| 優先度 | 基準 |
| --- | --- |
| `P0` | 受け入れ条件または契約を破り、作業を継続する前に判断が必要である。 |
| `P1` | 契約、主要な挙動、または安全性への影響が大きく、完了前に扱う必要がある。 |
| `P2` | 影響が局所的で、対応方法が明確である。 |
| `P3` | 影響が軽微である。 |

#### 判定後の扱い
- 妥当性を確定できない所見を `保留` にしない。
- 優先度は対応の緊急性を示すために使い、今回の対象範囲を拡張する根拠にしない。
- 採用所見が対象差分へ反映済みの場合は、判定を `採用` とし、対応方針を `反映済み / 追加対応なし` とする。

### Phase 4: 対応方針へ束ねる
- 採用所見を、同じ原因または変更で解消すべきまとまりに束ねる。
- 各まとまりに、内容を表すテーマ名、目的、対応方針、影響範囲、検証観点を付ける。
- `保留` は現在の対応方針へ混ぜず、別タスク候補へ分けて着手条件を残す。
- `要確認` は対応方針へ混ぜず、判定に必要な最小限の確認事項へまとめる。

### Phase 5: 結果を返す
`references/triage-report-template.md` に沿って、サマリー、所見別判定表、対応方針、別タスク候補、確認事項を返す。

## 完了条件
- すべての所見に `ID / 根拠 / 対象との関係 / 判定 / 優先度 / 対応方針` がある。
- 重複所見が統合され、`採用 / 却下 / 保留 / 要確認` が基準どおりに分かれている。
- 採用所見が対応方針へ束ねられ、保留所見と確認事項が分離されている。
