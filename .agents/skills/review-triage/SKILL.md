# 目的
- レビュー指摘を証拠ベースで評価し、事実誤認と有効な改善要求を混同しない。
- 重複した指摘を統合し、改善テーマ単位で設計へ束ねる。
- 次工程の種類を整理し、実装工程へ渡せる判断材料を残す。

# 非目的
- レビューシステムからコメントを自動取得すること。
- レビュー返信文や反論文を作成すること。
- コード編集、チケット更新、レビュー運用操作、テスト実行を行うこと。

# 入力
- レビューコメント本文。
- 対象コード、設計断片、または差分。
- 必要なら要求仕様、設計メモ、受け入れ条件、既存ドキュメントの文脈。
- 入力が粗い場合でも、まずリポジトリを探索して根拠を集める。証拠が足りない点だけを追加確認する。

# 出力
- `サマリー`
  - 全体結論、重複統合結果、主要リスク、未解決事項。
- `コメント別判定表`
  - 列は `ID / 要約 / 根拠 / 判定 / 優先度(P{N}) / 対応レーン / 改善方針` に固定する。
- `改善設計`
  - 採用項目を `契約整理 / 実装修正 / テスト補強 / ドキュメント更新 / 要件同期` のテーマへ束ねる。
  - 保留項目は follow-up 候補として分離する。

# 手順
## 1. 入力を整理する
- `assets/intake-template.md` に従って `背景 / レビュー目的 / コメント一覧 / 参照資料` を整理する。
- コメントごとに一意な `ID` を振る。

## 2. コメントを正規化し、重複を統合する
- 同じ failure mode、同じ契約違反、同じ観点を指す指摘を 1 件へ束ねる。
- 表現だけが違う重複は、最も具体的な指摘を代表として残す。

## 3. 証拠を集める
- `references/evidence-criteria.md` の優先順位に従い、仕様、契約、実装、既存テスト、文書を確認する。
- 根拠が見つからない点は推測で埋めず、不足情報として残す。

## 4. 正当性を判定する
- 各指摘を、集めた証拠と照合して `妥当 / 不妥当 / 証拠不足` のいずれかへ整理する。
- 根拠なしの好み指摘は、プロジェクト規約または明示仕様で裏づけられない限り採用しない。

## 5. 4分類トリアージを行う
- 各指摘へ次のいずれかを付与する。
  - `採用`: 現時点の証拠で妥当と判断できる指摘。
  - `却下`: 証拠と矛盾する、または誤読に基づく指摘。
  - `保留`: 妥当だが今回のレビュー対象境界外で、別タスクまたは後続レビューサイクルで扱うべき指摘。
  - `要確認`: 仕様、意図、前提が不足しており、この場で確定できない指摘。
- 優先度は `P0` から `P3` を使う。
  - `P0`: 受け入れ条件違反、契約破壊、重大な回帰リスク、または即時に結論が必要な論点。
  - `P1`: 今回のレビュー対象に強く関係し、現サイクルでの対応が望ましい論点。
  - `P2`: 局所的で回避策も明確だが、放置しないほうがよい論点。
  - `P3`: ドキュメント整理、軽微な可読性改善、後続でも安全に扱える論点。
- 対応レーンは次の 2 値だけを使う。
  - `current`: 今回のレビューサイクルで結論または反映が必要なもの。
  - `follow-up`: 別タスク、後続レビューサイクル、または後続の仕様確認として扱うもの。
- `保留` は「妥当だが今は扱わない」を意味する。妥当性が未確定なものは `要確認` にする。

## 6. 改善設計へ束ね直す
- 採用項目を次のテーマへ束ねる。
  - `契約整理`: 期待される振る舞い、境界条件、入力制約、戻り値保証の記述不足または不整合を直す。
  - `実装修正`: ロジック不備、境界条件漏れ、状態遷移不整合、例外処理不備を直す。
  - `テスト補強`: 契約は妥当だが、回帰を防ぐ証拠テストが足りない状態を補う。
  - `ドキュメント更新`: 実装は妥当だが、利用者やレビューアーが誤解しやすい説明不足を補う。
  - `要件同期`: 実装または合意済み方針が要求仕様や設計記録とずれている状態を揃える。
- 各テーマに `目的 / 対応方針 / 影響範囲 / 検証方法 / 次工程` を付ける。
- `次工程` には、テスト設計、ドキュメント更新、要件整理、最終検証など必要な作業種別だけを列挙する。
- `保留` は follow-up backlog へ分ける。
- `要確認` は改善設計へ入れず、先に必要な最小質問を列挙する。
- このスキル内では次工程を実行しない。

## 7. レポートを生成する
- `assets/triage-report-template.md` に沿って `サマリー / コメント別判定表 / 改善設計` を出力する。
- `要確認` が残る場合は、結論に必要な最小質問だけを併記する。

# 受け入れシナリオ
1. 事実誤認のレビュー指摘を、実装または仕様根拠に基づいて `却下` にできる。
2. 妥当な欠陥指摘を `採用` にし、同時に修正テーマと検証方法を提案できる。
3. 妥当だが今回のレビュー対象境界外の提案を `保留` にし、follow-up へ送れる。
4. 仕様または意図が不足している指摘を `要確認` にし、必要な確認事項を最小化できる。
5. 重複コメントを 1 件へ統合し、改善設計をテーマ単位へ束ねられる。
6. 差分が無い既存コードベースのレビューでも、対象コードと既存資料を根拠に判定と改善設計を作成できる。

# Definition of Done
- 全てのレビューコメントが `ID / 判定 / 根拠 / 優先度(P{N}) / 対応レーン` 付きで整理されている。
- 重複指摘が統合され、`保留` と `要確認` が `採用` から分離されている。
- `採用` 項目の改善設計に `テーマ / 検証方法 / 次工程` が含まれている。
- 実装変更やレビュー返信の作成へ逸脱していない。
