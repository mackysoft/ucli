# Evidence Criteria

## 根拠の優先順位
1. 明示仕様 / 合意済み要件 / 設計記録
2. 契約テスト / 公開インターフェース契約 / 公式ドキュメント
3. 実装の現挙動
4. レビュアーの推測または好み

上位根拠が下位根拠と矛盾する場合は、上位根拠を優先する。

## 証拠として採用するもの
- 合意済み要件、受け入れ条件、設計方針、Out of Scope
- 要件整理文書、設計メモ、公開仕様
- 公開インターフェースの契約、既存テスト、既存ドキュメント
- 対象コードから確認できる実際の制御フロー
- 差分がある場合は、その差分と周辺コードから確認できる変更意図と影響範囲

## 証拠として弱いもの
- 「一般にはこう書くべき」という一般論だけの指摘
- 対象コードやプロジェクト規約へ接続していない好み
- 実装を読まずに推測したパフォーマンス懸念

## 判定ルール
- 上位根拠に反する指摘は `却下` 候補とする。
- 上位根拠が未整備で、実装だけでは意図を確定できない指摘は `要確認` 候補とする。
- 指摘自体は妥当でも、今回のレビュー対象境界外で扱うべきものは `保留` 候補とする。
- 明示仕様がなくても、既存の契約テストや公開 API 契約と矛盾する指摘や提案は慎重に扱う。

## 好み指摘の扱い
- プロジェクト規約、既存の一貫した実装規則、または保守性上の明確な失敗モードが示せる場合のみ採用する。
- 裏づけがない場合は `却下` または `要確認` とし、好みを事実として扱わない。
