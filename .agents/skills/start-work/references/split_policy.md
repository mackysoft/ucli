# Recursive Split Policy（フレームワーク・ディレクトリ非依存）

## 目的
- 大きいタスクを「レビュー可能」「検証可能」「安全にマージ可能」な単位まで分割し、巨大PRを構造的に防ぐ。
- 分割の根拠を、固定パターンではなく **境界（Boundary）と依存関係（Dependency）**の分析で決める。
- 子タスクを作ったら同じ基準で再評価し、必要ならさらに分割する（再帰）。

---

## 基本概念（普遍）
### Boundary（境界）
責務・依存方向・契約が比較的安定している「分割の単位」。
境界は **ファイルパスで定義しない**。パスは“実装上の配置”に過ぎず、境界の本質は契約と依存にある。

境界の候補（一般形）：
- **ビルド単位**：パッケージ・モジュール・ライブラリ・プロジェクト
- **デプロイ単位**：アプリ・サービス・ジョブ・関数（単独で動く単位）
- **レイヤー単位**：UI・アプリケーション・ドメイン・インフラ など（名称はプロジェクト次第）
- **ツール・運用単位**：ビルド・生成・マイグレーション・CI・デプロイ設定（Runtimeとは別境界になりやすい）
- **テスト境界**：ユニット・統合・E2E、速度帯（Smoke/Regression）など

> 境界が曖昧な場合は、まず「依存方向」と「契約（API/データ）」を基準に仮の境界を切り、必要に応じて boundary map（後述）で明文化する。

### Contract（契約）
境界外から依存される形。
例：
- 公開API（関数・クラス・HTTP API・CLI）
- データ形式（DTO、永続化スキーマ、設定スキーマ、イベント形式）
- 例外・エラー契約、互換性条件

### Integration / Wiring（統合・配線）
境界同士をつなぐだけの変更。
例：
- DI / 登録 / ルーティング / 構成 / 初期化順 / 接続情報
- どのフレームワークでも存在する概念として扱う

## 分割の不変原則
- **1子タスクは、原則「主たる境界が1つ」**になるように作る。
  - 例外：Integration/Wiring-only の子タスクは許可（“配線だけ”に限定する）
- 各子タスクは **単独でマージ可能**（最低限のビルド・テストが通る）であること。
- 各子タスクには必ず以下を持たせる：
  - 触る境界（Boundary touched）
  - 変更種類（Contract / Implementation / Integration / Data）
  - 受け入れ条件（Acceptance Criteria）
  - 検証手段（どのテスト・どのチェックを通すか）
  - 依存関係（先に必要な子タスク）

## 再帰的分割アルゴリズム

### Step 0: 目的と外部可観測な結果を固定
- 「最終的に何が変わるか」を1文で書く（ユーザーから見える挙動・入出力・データの変化）。
- 受け入れ条件（チェックボックス）を列挙。
- スコープ外（やらないこと）を列挙。

### Step 1: 境界の列挙（Boundary scan）
- “触る/触りそうな境界”を列挙する。
- 境界の列挙は、次の順に行う（ディレクトリに依存しない）：
  1) ビルド単位（パッケージ・モジュール・プロジェクトなどの定義から拾う）
  2) デプロイ単位（サービス・アプリ・ジョブなど）
  3) レイヤー単位（依存方向の塊）
  4) ツール・運用単位（生成・マイグレーション・CI等）
  5) テスト境界（ユニット・統合・E2E）

### Step 2: 影響の分類（Impact classification）
各境界について、変更の種類を分類する：
- Contract change（契約が変わる・増える）
- Implementation change（内部実装）
- Integration change（配線・構成）
- Data change（データ形式・移行）

### Step 3: 境界跨ぎ（Cross-boundary edges）の抽出
次を満たすものを「境界跨ぎ」とする：
- 2つ以上の境界を同時に変更しないとコンパイル・実行が成立しない
- 依存方向を変える（または依存の追加で層が崩れる）
- データ形式変更により複数境界の同時更新が必要

境界跨ぎが多いほど、分割が必須。

### Step 4: 分割軸の選択（状況に応じて選ぶ）
固定パターンではなく、境界跨ぎが最小になる軸を選ぶ。

A) Contract-first（契約分離）
- 新しい契約を追加して互換を保ち、移行は後続に回す。

B) Parallel path / Strangler（並行経路）
- 旧経路を残して新経路を追加し、切替を最後に行う。
- 中間互換は許可するが、回収タスクが必須。

C) Boundary cut（境界ごとに切る）
- 境界ごとに子タスクを作り、他境界はスタブ・モック・アダプタで隔離する。

D) Use-case slice（縦割り最小スライス）
- 最小の価値（E2E最短経路）を先に通し、スライスを積む。

E) Risk spike（不確実性分離）
- 仕様・技術・再現性が不明な部分は調査タスクとして独立させ、結論をIssueに残す。

> 1つの子タスクに複数軸を混ぜない（混ぜるとレビュー不能になる）。

### Step 5: Change tree（子タスク群）を生成
- 親タスクは「計画（Plan）」として、子タスクに実装単位を落とす。
- 各子タスクは Step 1〜4 の分析結果を最小限で書く。

### Step 6: Leaf条件（停止条件）チェック → 満たさないなら再帰
子タスクが次を満たすまで分割を繰り返す：

Leaf（これ以上割らない）条件：
- 主たる境界が1つ（例外：Integration-only）
- 境界跨ぎが最小で、同時変更が不可避な理由が明文化されている
- 受け入れ条件が具体で、検証方法が定義できる
- 差分規模がレビュー可能（行数/ファイル数“だけ”ではなく、境界跨ぎの少なさを重視）
- 中間互換があるなら「回収タスク」が同一計画内に存在する

満たさない場合：
- その子タスクを Step 1 から再評価して、もう一段割る。

## 特殊ケース（普遍の扱い）
### 広範なAPI署名変更（多数呼び出しに波及）
- 旧を壊さず新を足す（Contract-first）
- 境界ごとに移行（Boundary cut）
- 最後に旧を削除（回収タスク）

### データ形式・スキーマ変更（永続化・設定・イベント）
- 追加→併存（読み取り互換/二重書き）→移行→削除、の段階を子タスク化
- 一括置換で巨大差分にしない

### 統合・構成・初期化（Integration/Wiring）が絡む
- 配線だけの子タスクとして分離するか、最終段に寄せる
- 変更が実行時にしか検証できない場合は、その検証を受け入れ条件に含める

## Waiver（分割免除）
分割免除は例外扱い。親タスク（Issue）に以下を必須で明記する：
- 免除理由（なぜ分割不能か）
- 境界跨ぎが不可避である根拠
- 追加の検証（テスト範囲拡大、レビュー人数増、段階リリース等）

明記が無い限り、免除は無効。
