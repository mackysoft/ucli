# Equivalence Rules (DRY)

## 統合判定キー
次の3条件が全て同じなら「本質的に同じテスト」と判定する。
- 契約観点
- 期待結果
- 失敗分類

## 必須ルール
- 同型ケースは統合を第一選択とし、パラメータ化で吸収する。
- 可読性のみを理由に重複を残さない。
- 同型判定が曖昧な場合は、契約観点の違いを先に明文化する。

## 優先順位
1. まず統合判定キーで同型かどうかを判定する。
2. 同型であっても、分離条件に該当する場合は分離を優先する。
3. 分離した場合は、理由をテスト計画またはレポートに残す。

## 分離が必要な条件
- 期待する契約が異なる。
- 失敗分類が異なる。
- 同一テストに統合すると診断可能性が下がる。

## 反パターン
- 入力だけ違う同一アサート列の複製。
- 既存ケースをコピーし、命名だけ変える。
