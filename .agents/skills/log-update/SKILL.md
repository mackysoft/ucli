# 目的
- 「いつ」「何を」「なぜ」「どう検証し」「次に何をするか」を短時間で再現できるログを残す。
- 作業の節目での更新を機械化し、ログ未更新の再発を止める。
- spec drift / 分割漏れ / 検証漏れが起きたとき、後から原因追跡できる粒度を確保する。

# 非目的
- 日記や実況（価値の低い逐次記録）は残さない。
- 仕様書（spec）やPR本文の代替をしない（ただしリンク・要点は書く）。
- 機密情報（トークン・鍵・個人情報）を記録しない。
- `log.md` が無い状態で新規作成すること。

# 使うタイミング
- 作業開始時（作業ブロックの開始）
- 方針・設計判断が確定した
- 主要な変更を入れた
- 検証を走らせた
- 失敗・詰まり・原因調査をした
- コミットを作った
- PRを作る直前（要点・リスク・検証結果のサマリを残す）
- 完了判断をした

# 契約
- 追記対象は `LOG_PATH` のみ（`LOG_PATH = docs/agents/<BRANCH_DIR>/log.md`）。
- 対象ファイルが無い場合はログ追記を Skip し、ファイル生成は行わない。
- `spec.md` / `log.md` の存在は StartWork 実行済み判定に使わない。

# 出力
- `LOG_PATH` への追記（存在時）
- または、ログ追記処理の Skip（不存在時）

# 手順
## 0. 追記対象を解決する（必須）
1. 現在ブランチ名を取得する。
2. `<BRANCH_DIR>` を算出する（ブランチ名の `/` を `_` に置換）。
3. `LOG_PATH = docs/agents/<BRANCH_DIR>/log.md` を設定する。
4. `LOG_PATH` の存在を確認する。
5. 存在しない場合は、ログ追記を Skip して終了する（ファイル作成はしない）。

## 1. 追記フォーマットを選ぶ
- 作業ブロック: `assets/entry_block.md`
- マイクロ更新: `assets/entry_micro.md`

## 2. 追記（append-only）
- 内容を、選んだフォーマットに沿って `LOG_PATH` に追記する。

## 記載スタイル
- 事実 → 判断 → 根拠（最小限）の順
- 「何をしたか」より「なぜそれをしたか」「どう検証したか」を優先
- 未確定は未確定として書く（決定事項にしない）

# Definition of Done
- `log.md` が存在する場合: トリガーに対応するエントリが append-only で追記されている
- `log.md` が存在しない場合: ログ追記処理が Skip され、ファイル生成を行っていない
- 機密情報が含まれていない
