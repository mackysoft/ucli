# コード品質の評価基準

## 重複

### 既存処理の再実装
- 既存の共通処理と同種の正規化、変換、判定、組み立てを別の場所で再実装していないかを見る。
- 実装形が違っても、入力契約と出力契約が実質同じなら候補に含める。

### 本質的な重複
- 名前や配置は違っていても、同じ分岐、同じ検証、同じ変換を複数箇所で持っていないかを見る。
- 重複により片方だけ修正される危険がある場合は優先度を上げる。
- 同じ条件を確認している場合は、各検証が所有する保証の対象、時点、成立範囲を比較する。違いを説明できなければ、保証の所有境界へ収束させる候補にする。

## 凝集性
- 一つの型、モジュール、関数が複数の変更理由を持っていないかを見る。
- 凝集は次の四つの観点で判断する。
  - 変更理由が同じか。
  - 同じ不変条件や同じ中核データを扱っているか。
  - 同じ処理フェーズに属しているか。
  - 同じ境界責務に属しているか。
- 次の混在は低凝集候補にする。
  - 方針決定と実行詳細が同居している。
  - 入力検証と正常系の本処理が密結合している。
  - データ変換と副作用が同じ単位に混在している。
  - 永続化、表示整形、通信、ドメイン計算が同じ単位で切り替わる。
  - 複数の操作種別やモードを、共有不変条件なしに一つの分岐群で抱えている。
- 次の兆候がある場合は、低凝集の可能性を上げる。
  - 一部のメンバや分岐が、同じ入力集合や同じ内部状態をほとんど共有していない。
  - 修正のたびに、同じ型やモジュールの別責務部分まで一緒に触る。
  - `private` ヘルパーや内部状態が、実質的に別々の責務ごとに固まっている。
- 一連の処理を順序づける責務、単一の境界責務、同じ不変条件を守る複数段階の処理は、複数の処理を含むことだけで低凝集と断定しない。
- 改善方向は、命名やファイル数ではなく、変更理由と不変条件の境界に合わせて責務を切り直すことを優先する。
- 機械的に細分化せず、分離によって呼び出し側の理解と変更影響が本当に小さくなる場合だけ分ける。

## 冗長な状態と中間表現
- 導出可能な値を重複保持していないか、意味を加えず受け渡すだけの中間オブジェクトが増えていないかを見る。
- 真偽値フラグの組み合わせでしか成り立たない状態表現や、寿命の短い一時構造が散らばっている場合は候補にする。

## 不要な抽象化と通過専用レイヤー
- 追加された抽象やレイヤーが、明確な契約差、差し替え理由、拡張点を持たず、既存処理の通過や名前の付け替えだけに留まっていないかを見る。
- 分離によって理解負荷だけが増え、呼び出し側の単純化、依存境界の明確化、変更容易性の向上に寄与していない場合は候補にする。

## 除外基準
- 契約が異なる。
- 境界が異なる。
- ライフサイクルや所有権が異なる。
- 性能、障害分離、並行性、配備単位の都合で分けている。
- 共通化よりも局所性と可読性を優先する方が明らかに有利である。

## 優先度

| 優先度 | 基準 |
| --- | --- |
| `P0` | 不整合、誤用、二重管理が広範囲へ波及し、直近の変更や運用を止める必要がある。 |
| `P1` | 同じ修正が複数箇所へ波及する、または境界を越えて設計負債が広がっている。 |
| `P2` | 現在の振る舞いは成立しているが、変更時の理解と修正の負担が高い。 |
| `P3` | 局所的な整理で改善でき、直近の不具合にはつながらない。 |
