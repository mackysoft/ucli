# Review Playbook

## 重点観点
### 1. Utility再実装
- 既存の共通処理と同種の正規化、変換、判定、組み立てを別の場所で再実装していないかを見る。
- 実装形が違っても、入力契約と出力契約が実質同じなら候補に含める。

### 2. 本質重複
- 名前や配置は違っていても、同じ分岐、同じバリデーション、同じマッピングを複数箇所で持っていないかを見る。
- 重複により片方だけ修正される危険がある場合は優先度を上げる。

### 3. 低凝集
- 1つの型、モジュール、関数が複数の変更理由を持っていないかを見る。
- 凝集は次の4軸で判断する。
  - 変更理由が同じか。
  - 同じ不変条件や同じ中核データを扱っているか。
  - 同じ処理フェーズに属しているか。
  - 同じ境界責務に属しているか。
- 次の混在は低凝集候補にする。
  - 方針決定と実行詳細が同居している。
  - 入力検証と正常系の本処理が密結合している。
  - データ変換と副作用が同じ単位に混在している。
  - 永続化、表示整形、通信、ドメイン計算が同じ単位で切り替わる。
  - 複数の操作種別やモードを、共有不変条件なしに1つの分岐群で抱えている。
- 次の兆候がある場合は、低凝集の可能性を上げる。
  - 一部のメンバや分岐が、同じ入力集合や同じ内部状態をほとんど共有していない。
  - 修正のたびに、同じ型やモジュールの別責務部分まで一緒に触る。
  - private helper や内部状態が、実質的に別々の責務ごとに固まっている。
- ただし次は低凝集と断定しない。
  - 1つのユースケースを順に進めるオーケストレーター。
  - 単一の境界責務に集中したアダプターやハンドラー。
  - 同じ不変条件を守るために複数段階の処理をまとめている単位。
- 改善方向は、命名やファイル数ではなく、変更理由と不変条件の境界に合わせて責務を切り直すことを優先する。
- 機械的に細分化せず、分離によって呼び出し側の理解と変更影響が本当に小さくなる場合だけ分ける。

### 4. 冗長状態・中間表現
- 導出可能な値を重複保持していないか、通過専用の state/context/request オブジェクトが増えすぎていないかを見る。
- 真偽値フラグの組み合わせでしか成り立たない状態表現や、寿命の短い一時構造が散らばっている場合は候補にする。

### 5. 不要抽象化・通過専用レイヤー
- 追加された抽象やレイヤーが、明確な契約差、差し替え理由、拡張点を持たず、既存処理の通過や名前の付け替えだけに留まっていないかを見る。
- 分離によって理解負荷だけが増え、呼び出し側の単純化、依存境界の明確化、変更容易性の向上に寄与していない場合は候補にする。

## 動作モード
- 評価専用:
  - 既定モードとして使う。
  - ユーザーの主目的がコードの状態把握、所見取得、レビュー依頼である場合に使う。
  - findings-first で返し、修正は行わない。
- 改善まで進む:
  - ユーザーが改善まで明示している場合、または上位フローが改善前提でこのスキルを呼んでいる場合に使う。
  - まず所見を作り、そのうち安全に扱えるものだけを自分で改善する。

## 除外条件
- 契約が異なる。
- 境界が異なる。
- ライフサイクルや所有権が異なる。
- 性能、障害分離、並行性、配備単位の都合で分けている。
- 共通化よりも局所性と可読性を優先する方が明らかに有利である。

## 優先度付け
- `P0`:
  - 放置コストが極めて高く、直近の変更や運用を止めるべき水準。
  - 不整合、誤用、二重管理が広範囲へ波及し、局所修正では抑え込めない。
- `P1`:
  - 同じ修正を複数箇所へ波及させる。
  - 誤用や不整合を生みやすい。
  - 境界横断で設計負債が広がっている。
- `P2`:
  - 現時点では壊れていないが、変更時の理解コストが高い。
  - 小さな差分のたびに周辺の複数要素を触る必要がある。
- `P3`:
  - 局所的な整理で改善できる。
  - すぐには不具合を生まないが、徐々に読みにくくなる。

## 報告順
### 評価専用
1. Findings
2. Open Questions or Assumptions
3. Summary

### 改善まで進む
1. Applied Improvements
2. Remaining Findings
3. Open Questions or Assumptions
4. Summary

## 所見の書式
- 各所見は `[P{N}]` を先頭に付ける。
- 優先度の後に、対象、問題の理由、分類、改善方向、根拠を続ける。

## 改善対象にしてよい条件
- 仕様変更を伴わない。
- 影響範囲が局所で、既存契約を壊さない。
- 追加のプロダクト判断や外部承認を要しない。
- 修正の意図と効果を説明できる。

## 改善対象にしない条件
- 大きな設計変更や責務再分割が必要で、妥当な分割案が複数ある。
- 正しさの判断に外部データ、運用事実、利用実績が必要である。
- 他の懸念よりも先に、仕様確定や受け入れ条件の確認が必要である。

## 所見なし時の扱い
- 無理に指摘を作らない。
- 「所見なし」を明示する。
- 未確認範囲、取得できなかった根拠、将来の変更で崩れやすそうな境界があれば残留リスクとして添える。
