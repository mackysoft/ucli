# code-quality-review

## 目的
コードベースまたは変更差分から、保守性を下げる重複、責務の混在、冗長な状態、不要な抽象化を特定する。
各所見の影響と根拠を示し、責務と契約を保った改善方向へ整理する。

## 参照
- 評価基準: [references/review-playbook.md](references/review-playbook.md)

## フロー
### Phase 1: レビュー範囲を確定する
- 依頼文に対象範囲が明示されていればそれを採用する。
- 明示が無ければ、まず変更差分を起点に確認し、必要に応じて周辺モジュールやコードベース全体へ広げる。
- 対象の要求と変更前から有効な契約を確認し、現状の構造だけを正しい設計の根拠にしない。
- 判断に必要な情報が不足する場合は、関連コード、履歴、設定、テスト、生成物から取得可能な根拠を集める。

### Phase 2: 構造を把握する
- 対象ファイルと、その呼び出し元、呼び出し先、関連する共通処理を読み、責務の境界を把握する。
- 共通化候補、状態保持箇所、変換処理、委譲だけの層を先に洗い出す。

### Phase 3: 候補を評価する
- `references/review-playbook.md` に従い、重複、低凝集、冗長な状態、不要な抽象化の候補を評価する。
- 名前や配置ではなく、責務、契約、データの流れ、変更理由の近さを根拠に候補化する。
- 凝集は、変更理由、共有不変条件、処理フェーズ、境界責務が揃っているかで判断する。
- 契約差異、境界差異、ライフサイクル差異、性能要件、障害分離の都合で分けている実装は、安易に統合候補にしない。
- 抽象化コストの方が高い小さな重複は、無理に所見へ昇格しない。

### Phase 4: 所見と改善方向を整理する
- 影響範囲、将来の変更コスト、誤用リスク、分岐の増殖傾向を見て優先度順に並べる。
- 各所見について、仕様変更なしで直せるか、影響範囲が局所か、追加の意思決定なしで整理できるかを判定する。
- 局所整理で済む所見、設計判断が必要な所見、別タスクへ送るべき所見を分ける。
- 各所見には、問題、保守性への影響、根拠、改善方向を対応させる。

### Phase 5: レビュー結果を返す
- 所見を優先度順に、`優先度 / 対象 / 問題 / 影響 / 根拠 / 改善方向` の順で提示する。
- 判断に必要な前提が不足する場合だけ、確認事項または仮定を補足する。
- 所見がない場合は、確認した範囲と残留リスクまたは未確認範囲を返す。

## 完了条件
- 対象範囲について、重複、凝集、状態、抽象化を確認している。
- すべての所見に優先度、対象、問題、影響、根拠、改善方向がある。
- 所見がない場合も、確認した範囲と残留リスクまたは未確認範囲が明らかになっている。
