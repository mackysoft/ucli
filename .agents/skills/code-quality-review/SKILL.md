# 目的
- コードベースの保守性を下げる重複、責務の混在、過剰な状態保持、不要抽象化を特定する。
- 指摘を列挙するだけでなく、どの方向に整理すべきかを返し、改善判断を前に進める。
- 自然文の依頼から対象範囲と重点観点を推定し、入力形式に依存しないレビューを行う。
- 評価結果を返して止まる場合と、そのまま安全な改善まで進む場合の両方で使えるレビュー起点を提供する。

# 使うタイミング
- 「この差分で既存処理を再実装していないか見たい」のように、重複や再発明を確認したいとき。
- 「全体を見て、構造的に危ない箇所だけ挙げたい」のように、コードベース品質を俯瞰したいとき。
- 凝集度、状態管理、抽象化の妥当性を、機能追加とは別軸でレビューしたいとき。
- 実装直後の自己チェックとして、品質上の問題を洗い出し、直せるものはそのまま改善まで持っていきたいとき。

# 使わないタイミング
- 主目的が新規仕様の実装そのもので、品質評価や改善判断を行う段階に入っていないとき。
- 仕様策定、要件整理、セキュリティ監査、性能測定が主目的であるとき。
- 外部環境、運用データ、本番観測など、このスキル単体では取得できない根拠が無いと判断できないとき。

# 入力（任意。無くても推定して進める）
- 対象範囲の希望
  - 例: 「この差分だけ」「このディレクトリ全体」「コードベース全体」
- 気になっている品質上の懸念
  - 例: 「重複実装」「状態が多すぎる」「抽象化が過剰」
- 期待する扱い
  - 例: 「評価だけしてほしい」「直せるものはそのまま直してよい」

# 出力
- 評価専用の場合:
  - findings-first のレビュー結果。
  - 各所見には、少なくとも `優先度 / 対象 / 問題の理由 / 分類 / 改善方向 / 根拠` を含める。
- 改善まで進む場合:
  - 適用した改善内容。
  - 修正後も残る所見だけを、`優先度 / 対象 / 問題の理由 / 分類 / 改善方向 / 根拠` 付きで返す。
  - 判断保留や未着手にした項目があれば、その理由を返す。
- 所見が無い場合、または改善で解消しきった場合は、その旨と残留リスクまたは未確認点を返す。

# 手順
## 1. 動作モードを決める
- ユーザーがレビュー、評価、所見出しを明示している場合は、評価専用として扱う。
- ユーザーが改善まで明示している場合、または上位フローが改善前提でこのスキルを呼んでいる場合は、改善まで進む前提で扱う。
- 明示が無い場合は、既定で評価専用として扱う。

## 2. レビュー範囲を確定する
- 依頼文に対象範囲が明示されていればそれを採用する。
- 明示が無ければ、まず変更差分を起点に確認し、必要に応じて周辺モジュールやコードベース全体へ広げる。
- 判断に必要な情報が不足している場合は、関連コード、履歴、設定、テスト、生成物など取得可能な根拠を追加で確認する。

## 3. 構造を把握する
- 対象ファイルと、その呼び出し元、呼び出し先、関連する共通処理を読み、責務の境界を把握する。
- 共通化候補、状態保持箇所、変換処理、委譲だけの層を先に洗い出す。

## 4. 候補を抽出する
- `references/review-playbook.md` の重点観点に沿って、重複、低凝集、冗長状態、不要抽象化の候補を拾う。
- 名前や配置ではなく、責務、契約、データの流れ、変更理由の近さを根拠に候補化する。
- 凝集は、変更理由、共有不変条件、処理フェーズ、境界責務が揃っているかで判断する。

## 5. 除外判定を行う
- 契約差異、境界差異、ライフサイクル差異、性能要件、障害分離の都合で分けている実装は、安易に統合候補にしない。
- 抽象化コストの方が高い小さな重複は、無理に所見へ昇格しない。

## 6. findings を整理する
- 影響範囲、将来の変更コスト、誤用リスク、分岐の増殖傾向を見て優先度順に並べる。
- 各所見は「何が冗長か」ではなく、「なぜ保守性を下げるか」と「どう整理するか」まで記述する。

## 7. 改善可能性を判定する
- 改善まで進むモードでは、各所見について、仕様変更なしで直せるか、影響範囲が局所か、追加の意思決定なしで安全に整理できるかを確認する。
- 上記を満たす所見は改善対象にする。
- 大きな設計変更、仕様判断、外部承認が必要なものは所見として残す。

## 8. 改善を適用し、再確認する
- 改善対象があれば優先度順に修正する。
- 修正後は対象範囲を見直し、同じ論点が解消したか、別の冗長性を生んでいないかを確認する。

## 9. レビュー結果を返す
- 評価専用の場合は、所見を優先度順に提示する。
- 改善まで進んだ場合は、適用した改善内容を先に要約し、残る所見だけを優先度順に提示する。
- 前提が不足して判断に幅がある場合だけ、Open Questions または Assumptions を補足する。
- 所見が無い場合、または改善で解消しきった場合は、確認した範囲と残留リスクまたは判断保留の理由を併記して終了する。

# Definition of Done
- 依頼文からレビュー範囲と動作モードを確定または推定し、その前提でレビューを完了している。
- 重複、凝集、状態管理、抽象化の観点を一通り確認している。
- すべての所見に `P{N} / 対象 / 問題の理由 / 分類 / 改善方向 / 根拠` が含まれている。
- 改善まで進むモードでは、安全に直せる所見は改善済みであり、残件には未着手理由がある。
- 所見が無い場合でも、「所見なし」と残留リスクまたは未確認点が返されている。
- すべての所見に `P{N}` 形式の優先度が付いている。
