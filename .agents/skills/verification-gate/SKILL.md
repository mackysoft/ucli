# verification-gate

## 目的
変更が受け入れ可能かを判断するために必要な検証を選び、実行し、結果を残す。

検証は技術スタック名ではなく、今回の作業が壊しうる性質から選ぶ。
検証対象がない、または判定材料にならない検証を増やさない。

## フロー

### Phase 1: 作業種別を固定する
変更内容、差分、依頼文、既存の受け入れ条件から、今回の作業種別を決める。

| 作業種別 | 主な確認対象 |
| --- | --- |
| ソースコードの挙動変更 | ビルド、振る舞い、静的解析 |
| API、CLI、設定、スキーマ変更 | 契約、互換性、ドキュメント |
| 生成物、テンプレート、スキル、文書変更 | 生成、スキーマ、差分整合性、表示 |
| CI、ビルド、パッケージ、リリース変更 | パイプライン、パッケージ、インストーラ、公開前確認 |
| UI、表示、レイアウト変更 | 表示、スナップショット、操作 |
| テストのみの変更 | 変更したテスト、テスト意図、関連する振る舞い |
| リファクタリング | ビルド、関連する振る舞い、静的解析 |

複数の作業種別にまたがる場合は、それぞれの確認対象を統合する。

### Phase 2: 検証面を選ぶ
受け入れ判定に必要な検証面だけを選ぶ。

| 検証面 | 確認する性質 |
| --- | --- |
| 機械的整合性 | コード整形、静的検査、生成物の更新漏れ、スキーマ、リンク、相互整合性 |
| ビルド | コンパイル、型検査、バンドル、アセット読み込み、パッケージ作成 |
| 振る舞い | 単体テスト、結合テスト、E2E テスト、スナップショットテスト、ゴールデンテスト |
| 契約 | API、CLI、設定、ファイル形式、出力形式、互換性 |
| 解析 | 静的解析、アーキテクチャ検査、セキュリティ検査、カバレッジ、品質指標 |
| 表示 | 画面、文書、画像、レイアウト、表示崩れ |
| リリース | パッケージ検証、インストール検証、公開前試行、移行検証 |

ユーザーが検証を明示した場合は、その検証面を含める。
受け入れ条件または既存規約で検証が求められている場合は、その検証面を含める。
選んだ検証面ごとに、判定する受け入れ条件または既存規約を対応させる。
検証で用いる入力、実行環境、準備、観測条件は、対応する受け入れ条件を測るための条件として扱う。受け入れ条件に含まれない条件を、検証結果から変更対象の契約や規約へ追加しない。
検証結果が今回の受け入れ判定に影響しない場合は実行しない。

### Phase 3: 実行コマンドを決める
実行コマンドは次の順に決める。

1. リポジトリの専用スクリプト。
2. CI で使われているコマンド。
3. `README`、開発文書、`AGENTS.md` に書かれた検証手順。
4. パッケージ、プロジェクト、ワークスペースの設定から分かる標準コマンド。
5. それでも決められない場合は、検証できない理由を残す。

複数の同等コマンドがある場合は、CI と同じものを優先する。
対象を限定できる場合は、受け入れ判定に必要な範囲へ限定する。

### Phase 4: 検証する
選んだ検証面ごとにコマンドを実行する。

#### 実行結果

| 状態 | 扱い |
| --- | --- |
| コマンドが成功した | 検証面ごとに成功として記録する。 |
| コマンドが失敗した | 失敗した検証面、コマンド、失敗理由、次に必要な作業を残す。 |
| コマンドが成果物を更新した | 検証は続けてよいが、未コミット差分として扱う。 |
| コマンドを決められない | 代替できる確認と、残る未検証範囲を残す。 |

#### 再実行
失敗時に、原因不明のまま同じコマンドを繰り返さない。
再実行は、原因に関係する入力、前提、実装、検証手段が変わった場合、または再現性の確認が受け入れ判定に必要な場合に限る。

#### 失敗の分類
変更対象の契約違反と判断するのは、根拠が確定している受け入れ条件または既存規約を破った場合に限る。
対応する条件を示せない失敗は、検証手段の不備、検証の前提不足、または今回の変更対象外で発生した問題として扱う。
検証を成功させるために、変更対象の契約や対応範囲を狭めない。

| 分類 | 扱い |
| --- | --- |
| 変更対象の契約違反 | 変更対象を修正する必要がある状態として残す。 |
| 検証手段の不備 | 検証方法または判定条件の修正が必要な状態として残す。 |
| 検証の前提不足 | 前提を整える必要がある状態、または未検証として残す。 |
| 今回の変更対象外で発生した問題 | 受け入れ判定から分離し、必要な場合だけ別の作業として残す。 |

### Phase 5: 判定と状態を残す
実行した検証、成功、失敗、未実行、未検証範囲を残す。

| 判定 | 条件 |
| --- | --- |
| 成功 | 受け入れ判定に必要な検証が成功し、未コミット差分がない。 |
| 未完了 | 検証は成功したが、生成物更新や整形などの未コミット差分が残っている。 |
| 失敗 | 必要な検証が失敗している。 |
| 未検証あり | 必要な検証を決められない、または実行できない範囲が残っている。 |

報告には、検証面、実行コマンド、結果、未検証範囲、次に必要な作業を含める。
失敗した場合は、修正に進むのではなく、原因分類と次の作業を残す。
