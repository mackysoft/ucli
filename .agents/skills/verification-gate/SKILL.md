# 目的
- PR前に通すテストゲートを固定し、テスト漏れを防ぐ。
- 今回の成果の受け入れ判定に必要なテスト範囲だけを選ぶ。

# 使うタイミング
- 変更が一通り終わった際の挙動確認
- PRを作る前のテスト
- 受け入れ条件を満たすかテストで確認したいとき

# 使わないタイミング
- まだ設計が固まっていない、または仕様が未確定
- 変更途中で、テスト結果を判定材料にできない（ただし Smoke は可）

# 契約
- テストは受け入れ判定に必要な範囲だけ実行する。
- 仕様にない確認を追加しない。
- テスト対象を絞る場合は、理由を出力へ残す。

# 参照
- テスト範囲: [references/gate_levels.md](references/gate_levels.md)
- 失敗分類: [references/failure_triage.md](references/failure_triage.md)

# 入力（任意。無くても推定して進める）
- 目的（優先）:
  - 「挙動確認」 / 「PR前」 / 「最終」など
- 希望のゲートレベル（任意）:
  - `Smoke` / `Standard` / `Full`
- テスト実行方針（任意）:
  - `Auto` / `Force`
- 影響が大きいと感じる点（任意）:
  - 例: 「コアロジック変更」「データ形式変更」「UIフロー変更」など
- テスト対象（任意）

# 出力
- 実行したテストコマンドと結果
- テスト範囲の判定（Run/Skip）と判定理由
- 失敗時: 失敗分類と次の最短リトライ方針

# 手順
## 1. ゲートレベルを決める（必須）
- ユーザーが明示した場合: そのレベルを採用
- 明示が無い場合: 目的で決める
  - 「PR前」→ Standard（原則）
  - 「最終」→ Full
  - 「挙動確認」→ Smoke（原則）

## 2. テスト実行要否を判定する（必須）
1. `テスト実行方針 = Force` の場合は、テストを `Run` にする。
2. `Auto`（既定）の場合、次を満たすときのみ `Run` にする。
   - ユーザーがテスト実行を明示している
   - 受け入れ条件または合意済み仕様にテスト要件がある
   - テストを実行しないと、今回成果の受け入れ可否を判定できない
3. 上記を満たさない場合は `Skip` とし、理由を出力へ残す。

## 3. テストゲートを実行する
1. `Skip` 判定の場合は、テストを実行せず最終判定へ進む。
2. `Run` 判定の場合は、`references/gate_levels.md` に従って実行範囲を決める。
3. 対象に適合するテストランナーを選択してテストを実行する。
4. 失敗した場合:
   - `references/failure_triage.md` で分類し、「最短の再実行方針」と「修正の当たり」を提示する
   - ゲートは FAIL

# 最終判定（PASS/FAIL）
- FAIL 条件:
  - `Run` 判定になったテストが失敗した
- PASS 条件:
  - `Run` 判定になったテストが成功した
  - または、テストが `Skip` 判定で、理由が受け入れ判定に対して説明できる

# Definition of Done
- 実行したテストコマンドと結果が出力として残っている
- テストの Run/Skip と理由が残っている
- FAIL の場合は次のアクション（修正 / 再実行方針）が明記されている
